Attention: how one token reads the room

Warm-up — recall from Lesson 2 (don't peek) In Lesson 2 we said the word "bank" starts as one static row from the embedding table, then comes out the top of the model as a contextual embedding that differs between "river bank" and "bank loan". We waved a hand and said "attention mixes in the neighbors." This lesson is that hand-wave, made mechanical.

Here's the whole idea in one sentence: attention lets each token pull in information from the other tokens, weighted by how relevant each one is. Every token gets to look around at the sequence and decide, "which of these others should reshape me?" — and then it blends them in. That reshaping is the contextual embedding. Landing how the weighting works is today's win.

The mental model: a soft dictionary lookup

A normal dictionary lookup is hard: you hand it a key, it returns the one value that matches exactly. Attention is the soft version — every entry matches a little, and you get back a blend of all their values, weighted by how well each matched.1 Three roles make it work, and each token produces all three:

RoleQuestion it answersDictionary analogy
Query "What am I looking for?"the key you type in
Key "What do I have to offer?"each entry's label
Value "What will I contribute if picked?"each entry's content
Anchor to what you know Where do Q, K, V come from? Each is the token's embedding pushed through its own learned linear layer — three weight matrices WQ, WK, WV, trained by backprop. So Q/K/V are just three linear projections of the same vector — nothing you haven't seen; the model learns three different "views" of each token.

The mechanism, in four steps

Take the token "bank". It emits a query. To decide how much each other token should reshape it:

  1. Score. Dot the query with every token's key. A big dot product = "these point the same way" = relevant.
  2. Scale. Divide each score by √dk (more on this below).
  3. Softmax. Turn the scores into weights that are positive and sum to 1 — a set of attention weights, one per token.
  4. Blend. Take the weighted sum of every token's value. That blended vector is "bank"'s new, context-aware representation.

Those four steps are the entire mechanism. Written compactly, they are one line — the famous scaled dot-product attention equation:

Attention(Q, K, V) = softmax( Q·Kᵀ / √dk ) · V

Read it left to right and it's exactly the four steps: Q·Kᵀ scores every query against every key, /√dk scales, softmax makes weights, and ·V blends the values. Every token does this at once, so one matrix multiply reshapes the whole sequence in parallel.2

Why the ÷√dₖ? Dot products of long vectors get large as dimension grows, and huge scores push softmax into a corner — one weight ≈ 1, the rest ≈ 0, and the gradient nearly vanishes so learning stalls. Dividing by √dk keeps the scores' spread roughly constant regardless of head size, so softmax stays sensitive. It's a stability trick, not a conceptual pillar.2
The practical payoff — why long context is expensive and "lost in the middle" happens Every token queries every other token: for n tokens that's n² score computations. Double the prompt and attention work quadruples — this is the core reason long context costs more and runs slower, and why context limits exist at all.3 It also hints at "lost in the middle": attention is a competition for a fixed budget of weight (softmax sums to 1), so the more tokens compete, the easier it is for the middle of a long prompt to get starved. You can now reason about both from the mechanism, not folklore.

The one-sentence win

You can now explain a contextual embedding mechanically: each token emits a query, scores it against every token's key, softmaxes those into attention weights, and blends the values — softmax(QKᵀ/√dₖ)·V — and you know that "attends to everything" is why long context is quadratically expensive.
