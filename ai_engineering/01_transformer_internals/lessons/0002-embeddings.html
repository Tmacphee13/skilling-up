Embeddings: three things everyone calls the same word

Warm-up — recall from Lesson 1 (don't peek) Your prompt reached the model as a list of integer token IDs like [9906, 11, 1917]. But a neural net multiplies matrices — it can't do math on the label "9906". So how does an integer ID become numbers the network can compute with? Hold your answer; this lesson is the bridge.

The answer is an embedding: a vector of numbers that stands in for a token. The trouble is that "embedding" is used for three genuinely different things, and people conflate them constantly. Untangling them is today's win — and it's the difference between understanding a model's guts and understanding the embeddings API you call for search.

Type 1 — Token embeddings (the lookup table)

This is the bridge from Lesson 1. Inside the model sits one big matrix: the embedding table, with one row per vocabulary entry — shape [vocab_size × d_model], e.g. [100000 × 2048]. To embed token ID 9906, you just grab row 9906. That's it. ID in, vector out.

Anchor to what you know This table is learned by backprop, exactly like any other weight matrix you know. A lookup-by-ID is mathematically identical to multiplying a one-hot vector by a linear layer — so a token embedding is just a linear layer you already understand, indexed instead of multiplied.

Because they're learned, the rows arrange themselves so that direction encodes meaning. The famous demonstration: vec("king") − vec("man") + vec("woman") ≈ vec("queen").1 Similar words land near each other; consistent relationships become consistent directions.

But note the ceiling: a token embedding is static. The word "bank" gets one row, the same vector whether you meant a river or a vault. Context hasn't entered yet.

Type 2 — Contextual embeddings (the model's output)

Now run those static vectors up through the transformer's layers. At each layer, attention lets every token mix in information from its neighbors (that's Movement II — we're foreshadowing). What comes out the top is a contextual embedding: a vector for each token that has been reshaped by the specific sentence around it.

Now "bank" in "river bank" and "bank" in "bank loan" get different vectors. Same starting row, different endings — because attention folded in different neighbors. This is the entire point of the architecture, previewed in one word.

Type 3 — Sentence embeddings (one vector per text)

Sometimes you don't want a vector per token — you want one vector for a whole sentence or document, so you can measure how similar two texts are. That's a sentence embedding. It's typically made by pooling the contextual embeddings (e.g. averaging them) — often from a model fine-tuned specifically for the job.

The practical payoff — this is what your embeddings API returns When you call an embeddings endpoint for semantic search or RAG, you get back Type 3: one vector per chunk. You store them and rank by cosine similarity (angle between vectors) to find text with similar meaning. The "embeddings" in your vector database are not the token-embedding table buried inside the model — a distinction almost everyone blurs.

The one-sentence win

You can now separate the three "embeddings" people constantly conflate: a static lookup row (token), the context-mixed vector the model outputs (contextual), and the one-vector-per-text you store for search (sentence) — and you know your RAG pipeline uses the third, not the first.
